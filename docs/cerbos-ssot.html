02 — The building blocks

Products, user levels, user types & resources

Every access decision combines four things: who the user is (level × type → a role), what products they've subscribed to, which resource they're touching, and which retailer they belong to.

Products — a subscription gate; a resource is invisible unless its product is on the account

“dealdesk” is not a product. It's a resource group, gated per-surface (retailer path): each resource requires the specific product for its surface — ssp, trade, or brand_center — while cross-cutting resources accept any one of the three and sites/blueprints are ungated (base). See Special Cases for the full breakdown and the brand-user exception.

User Levels userLevel

The organizational tier — decides scope. Higher tiers reach into lower ones.

  • su — Super User / platform. Reaches everything; not bound to one retailer.
  • agency — Manages its own child retailers only.
  • retailer — The operating store level. The scope floor — cannot delegate.
  • brand — DealDesk-only, cross-retailer (scoped to linked retailers).

User Types userType

The seniority within a level. Level × Type computes the actual role.

  • owner — Full control.
  • admin — Manage users/resources & settings.
  • lead — Operate resources; no settings access.
  • member — Standard operator.
  • collaborator — Read-only (list / view / export).

Roles are derived 4 × 5

Roles aren't assigned directly — Cerbos computes them from level × type. Naming isn't always parallel:

  • retailer + admin → retailer_manager
  • retailer + lead → team_lead
  • retailer + member → staff_operator
  • retailer + collaborator → guest_collaborator
  • Brand level has no collaborator (4 types only).

04 — Operating guide

How-to: permission-types & seeding

Two recurring jobs: consuming the shared @conqrse/permission-types package, and pushing policies to the Cerbos servers. The matrix doc is the source — policies and types are both generated from it.

@conqrse/permission-types v1.7.0

Auto-generated TypeScript enums & types shared by conqrse-admin and conqrse-api3. Never hand-edit — it's generated from the matrix.

install in a consumer
# consumer .npmrc must point the scope at the private registry
@conqrse:registry=https://npm.conqrse.com/

npm install @conqrse/permission-types@latest
use it
import { Resource, Action, RESOURCE_META } from '@conqrse/permission-types';

const meta = RESOURCE_META[Resource.QR_CAMPAIGNS];
meta.type;      // 'collection'
meta.actions;   // [Action.LIST, Action.VIEW, ...]
meta.products;  // [Product.QR, ...]

Regenerate & publish on policy change

After the matrix changes, regenerate the package, bump the version, and publish. A version bump is mandatory — if enums drift from deployed policies, runtime auth breaks.

regenerate → build → publish
# 1. regenerate enums from the matrix
python3 scripts/generate_types.py --force

# 2. build
cd packages/permission-types && npm run build

# 3. bump version in package.json (semver):
#    added value = minor · removed/renamed = major · fix = patch

# 4. publish + verify
npm publish --registry https://npm.conqrse.com/
npm view @conqrse/permission-types@latest \
  --registry https://npm.conqrse.com/

Seeding Cerbos policies — staging then production

Seeding is a Kubernetes Job (cerbos-seed-policies) driven by k8s/deploy.sh. The script compiles the policies, re-applies the kustomize overlay, and re-runs the seed job — which POSTs every file in the cerbos-policies ConfigMap to the Cerbos admin API. Same script, same job — the only difference is the environment argument.

./k8s/deploy.sh staging

kubectl get job -n staging cerbos-seed-policies
kubectl logs -n staging job/cerbos-seed-policies | tail -20

kubectl get job -n production cerbos-seed-policies

✓
A healthy seed ends with Successfully seeded <N> policies where N matches the ConfigMap file count, no ERROR: lines, and your files listed as Seeded: resource_<name>.yaml.

When the matrix changes — end-to-end

  1. Edit docs/RESOURCES_ACTIONS_MATRIX.md (resource-actions table + Product × Resource matrix; required = OR. DealDesk is gated per-surface by namespace; required-all = AND is deprecated).
  2. Add the new policy filename to k8s/base/kustomization.yaml under configMapGenerator.files. A file not in this allowlist never reaches any cluster.
  3. Preview, then generate: python3 scripts/generate_policies.py --dry-run → --force (or --resource "<name>" --force).
  4. Regenerate types: python3 scripts/generate_types.py --force, then build & bump the package.
  5. Validate locally: cerbos compile k8s/base/policies/.
  6. Commit, then deploy staging → production using the commands above.
  7. Refresh the status doc: python3 scripts/generate_status.py.

06 — Read before you rely on it

Special cases & gotchas

The rules below are not obvious from the resource names. They shape who really gets access.

🔐 Retailer isolation

Every :item (single-record) product resource enforces P.attr.retailerId == R.attr.retailerId — the user's retailer must equal the record's retailer. This blocks cross-retailer data access. SU is the only level not bound to a single retailer; agencies are bound to their own child retailers. List endpoints don't always carry a retailerId, so the consuming API must filter lists to the caller's retailer.

🪪 Delegation / “act-as”

SU and Agency levels reach into Retailer resources with the same permissions as retailer owners/managers — delegation is achieved implicitly through userLevel scope, not a separate mechanism. Retailer level cannot delegate — it cannot act as another retailer. Agencies are bounded to their own child retailers. Note: the two test docs disagree on whether “Act-As” is still a distinct pattern — treat it as scope-based.

🤝 DealDesk gates per-surface

dealdesk is a resource group, not a product. As of 2026-07-16 the retailer path gates per-surface (the old all-three AND is deprecated): ssp → ssp, dsp-blocks; trade → trade-ledgers, media-packages, line-items, photo-verifications; brand_center → brands, brand-users. Cross-cutting resources (campaigns, inventory, reports, placement-rules, placement-types, rate-cards) accept any one of the three; store-traffic needs ssp or trade; sites and blueprints are base (no product gate). Each policy also keeps a separate brand path (userLevel == "brand") that only needs brand_center and is scoped cross-retailer via R.attr.retailerId in P.attr.retailerIds (the brand's linked retailers).

👁️ Collaborators are read-only

At every level, collaborator users may only list / view / export. Any create / update / delete is denied. Separately, lead-type users have no settings access — only owner/admin can change settings.

🧩 The allowlist is the shipping gate

A resource_*.yaml file only ships if it's listed in k8s/base/kustomization.yaml. This is the single most common cause of “I deployed but nothing changed.” 18 CMS policies exist in the repo but are absent from the allowlist — so CMS authorization is defined but will never reach any cluster.

🗂️ Settings are non-product

settings:admin_* and settings:user_* do not require product or retailer checks — they're cross-cutting. Note the naming: settings use underscores (settings:admin_users), deliberately distinct from product-namespaced resources.

🧬 Legacy brand_user role

A generic brand_user role (matches any userLevel == "brand") is kept only for backward compatibility with dealdesk_brand:media-packages, and is slated for removal. The legacy dealdesk_brand:* grouping behaves differently from the newer tiered dealdesk:* policies.

📦 Version bump is mandatory

Every policy change must bump @conqrse/permission-types. Consumers rely on the generated enums; if deployed policies and enums diverge, runtime authorization breaks. Never overwrite a published version — always bump.
